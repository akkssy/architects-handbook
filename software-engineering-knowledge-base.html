üìê SOLID Principles in Architecture Context Foundation

Understanding SOLID at the architectural level is crucial for senior engineers and architects:

Single Responsibility Principle (SRP) - Service Level

Each microservice should have one reason to change. A service handling both user authentication AND user notifications violates SRP.

C# Example - SRP Violation vs Correct
// ‚ùå VIOLATION: Service doing too much
public class UserService
{
    public async Task<User> AuthenticateUser(string email, string password);
    public async Task SendWelcomeEmail(User user);
    public async Task GenerateReport(User user);
}

// ‚úÖ CORRECT: Separate services with single responsibility
public class AuthenticationService { /* Auth only */ }
public class NotificationService { /* Notifications only */ }
public class ReportingService { /* Reports only */ }

Open/Closed Principle (OCP) - Extension Points

Design systems that can be extended without modification. Use plugin architectures, strategy patterns, and event-driven designs.

C# Example - Plugin Architecture
// Define extension point
public interface IPaymentProcessor
{
    string ProviderName { get; }
    Task<PaymentResult> ProcessPayment(PaymentRequest request);
}

// New payment providers can be added without changing existing code
public class StripeProcessor : IPaymentProcessor { }
public class PayPalProcessor : IPaymentProcessor { }

// Registration via DI - open for extension
services.AddScoped<IPaymentProcessor, StripeProcessor>();
services.AddScoped<IPaymentProcessor, PayPalProcessor>();

Dependency Inversion Principle (DIP) - Service Contracts

High-level modules (business logic) should not depend on low-level modules (data access). Both should depend on abstractions.

Architecture Impact: DIP leads to better testability, easier service replacement, and cleaner boundaries between layers.

‚öñÔ∏è CAP Theorem & Distributed Systems Essential

In a distributed system, you can only guarantee two of three properties:

Consistency (C)

All nodes see the same data at the same time

Availability (A)

Every request receives a response (success/failure)

Partition Tolerance (P)

System continues operating despite network failures

Real-World Trade-offs

System Type Choice Example
Banking/Financial CP (Consistency + Partition Tolerance) Account balances must be accurate
Social Media AP (Availability + Partition Tolerance) Posts can be eventually consistent
E-commerce Cart AP with conflict resolution Cart merging on reconnection
Modern Reality: Most systems use a spectrum approach. PACELC theorem extends CAP: "if Partition, choose Availability or Consistency; Else (normal operation), choose Latency or Consistency."

üîó Design: URL Shortener (like bit.ly) Classic

1. Requirements Clarification

Functional:

  • Shorten a long URL to a short URL
  • Redirect short URL to original URL
  • Optional: Custom aliases, expiration, analytics

Non-Functional:

  • High availability (99.9%)
  • Low latency redirects (<100ms)
  • Short URLs should be unpredictable

2. Estimations

Back-of-Envelope Calculations
// Assumptions
New URLs per month: 100M
Read:Write ratio: 100:1

// QPS Calculations
Writes: 100M / (30 * 24 * 3600) ‚âà 40 writes/sec
Reads: 40 * 100 = 4,000 reads/sec

// Storage (5 years)
Records: 100M * 12 * 5 = 6 billion records
Size per record: ~500 bytes
Total: 6B * 500 bytes = 3 TB

3. Data Model

SQL Schema
CREATE TABLE urls (
    id BIGINT PRIMARY KEY,
    short_code VARCHAR(7) UNIQUE NOT NULL,
    original_url VARCHAR(2048) NOT NULL,
    user_id BIGINT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP,
    click_count BIGINT DEFAULT 0
);

CREATE INDEX idx_short_code ON urls(short_code);

5. Key Design Decisions

Short Code Generation Options:
  • Base62 encoding: Convert auto-increment ID to [a-zA-Z0-9] = 62^7 = 3.5 trillion combinations
  • Pre-generated keys: Generate keys in advance, store in separate KV store
  • Hash-based: MD5/SHA256 of URL, take first 7 characters (collision handling needed)
C# - Key Generation Service
public class KeyGenerationService
{
    private const string Alphabet = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";

    public string GenerateShortCode(long id)
    {
        var sb = new StringBuilder();
        while (id > 0)
        {
            sb.Insert(0, Alphabet[(int)(id % 62)]);
            id /= 62;
        }
        return sb.ToString().PadLeft(7, 'a');
    }
}

Key Components

C# - Notification Request Handler
public class NotificationRequest
{
    public string UserId { get; set; }
    public string TemplateId { get; set; }
    public Dictionary<string, object> Data { get; set; }
    public NotificationChannel[] Channels { get; set; }
    public Priority Priority { get; set; } = Priority.Normal;
}

public class NotificationService
{
    private readonly IServiceBusClient _serviceBus;
    private readonly IUserPreferenceRepository _prefs;

    public async Task<NotificationResult> SendAsync(NotificationRequest request)
    {
        // 1. Validate user preferences
        var userPrefs = await _prefs.GetAsync(request.UserId);
        var allowedChannels = FilterByPreferences(request.Channels, userPrefs);

        // 2. Queue messages for each channel
        foreach (var channel in allowedChannels)
        {
            await _serviceBus.SendAsync(
                queueName: $"notifications-{channel}",
                message: new ChannelMessage(request, channel)
            );
        }

        return new NotificationResult { Status = Status.Queued };
    }
}
Scale Considerations: At high scale, use separate queues per priority level. Critical notifications (password reset) should never be delayed by marketing emails.

Workflow Engine Architecture

C# - Workflow Engine Core
public interface IWorkflowEngine
{
    Task<string> StartProcessAsync(string processDefinitionId, Dictionary<string, object> variables);
    Task CompleteTaskAsync(string taskId, Dictionary<string, object> variables);
    Task SignalEventAsync(string processId, string eventName);
}

public class WorkflowEngine : IWorkflowEngine
{
    private readonly IProcessRepository _processRepo;
    private readonly ITaskExecutor _taskExecutor;
    private readonly IEventBus _eventBus;

    public async Task<string> StartProcessAsync(
        string processDefinitionId,
        Dictionary<string, object> variables)
    {
        // 1. Load process definition (BPMN)
        var definition = await _processRepo.GetDefinitionAsync(processDefinitionId);

        // 2. Create process instance
        var instance = new ProcessInstance
        {
            Id = Guid.NewGuid().ToString(),
            DefinitionId = processDefinitionId,
            State = ProcessState.Active,
            Variables = variables,
            CurrentTokens = new List<Token> { new Token(definition.StartEvent) }
        };

        // 3. Execute until wait state
        await ExecuteAsync(instance);

        return instance.Id;
    }

    private async Task ExecuteAsync(ProcessInstance instance)
    {
        while (instance.HasExecutableTokens)
        {
            foreach (var token in instance.GetExecutableTokens())
            {
                var element = token.CurrentElement;

                switch (element)
                {
                    case ServiceTask st:
                        await _taskExecutor.ExecuteServiceTaskAsync(st, instance);
                        token.MoveTo(st.OutgoingFlow.Target);
                        break;

                    case UserTask ut:
                        // Wait state - create human task
                        await CreateHumanTaskAsync(ut, instance);
                        token.State = TokenState.Waiting;
                        break;

                    case ExclusiveGateway xg:
                        var outflow = EvaluateGateway(xg, instance.Variables);
                        token.MoveTo(outflow.Target);
                        break;

                    case ParallelGateway pg:
                        HandleParallelGateway(pg, token, instance);
                        break;

                    case EndEvent:
                        instance.CompleteToken(token);
                        break;
                }
            }
        }

        await _processRepo.SaveAsync(instance);
    }
}

public class OrderMessageHandler
{

public record CreateOrderCommand(
    string CustomerId,

// Command Handler
public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, OrderResult>
{
    private readonly IEventStore _eventStore;

    public async Task<OrderResult> Handle(
        CreateOrderCommand request,
        CancellationToken ct)
    {

        var order = new Order(request.CustomerId);
        foreach (var item in request.Items)

// Anti-Corruption Layer translates between legacy and new systems
public class CustomerAntiCorruptionLayer
{
    private readonly ILegacyCustomerService _legacy;
    private readonly INewCustomerService _new;
    private readonly IFeatureFlags _features;

    public async Task<CustomerDto> GetCustomerAsync(string id)
    {
        // Feature flag controls routing
        if (await _features.IsEnabledAsync("UseNewCustomerService", id))
        {
            var customer = await _new.GetAsync(id);
            return MapToDto(customer);
        }

        // Legacy path - translate old format to new
        var legacyCustomer = await _legacy.RetrieveCustomer(id);
        return TranslateLegacyToDto(legacyCustomer);
    }

    private CustomerDto TranslateLegacyToDto(LegacyCustomerRecord legacy)
    {
        // Handle legacy data quirks
        return new CustomerDto
        {
            Id = legacy.CUST_ID.Trim(),
            Name = $"{legacy.FIRST_NM} {legacy.LAST_NM}".Trim(),
            Email = legacy.EMAIL_ADDR?.ToLowerInvariant(),
            Status = MapLegacyStatus(legacy.STAT_CD)
        };
    }
}

How would you design a rate limiter for an API?

Approach Framework

  1. Clarify requirements: Rate per user? Per IP? Per API key? Global or per-endpoint?
  2. Choose algorithm: Token Bucket, Leaky Bucket, Fixed Window, Sliding Window
  3. Storage: Redis for distributed rate limiting

Recommended: Sliding Window Counter

public class SlidingWindowRateLimiter
{
    private readonly IDistributedCache _cache;

    public async Task<bool> IsAllowedAsync(string clientId, int maxRequests, TimeSpan window)
    {
        var now = DateTimeOffset.UtcNow;
        var windowStart = now.AddSeconds(-window.TotalSeconds);

        // Get current and previous window counts
        var currentKey = $"rate:{clientId}:{now.ToUnixTimeSeconds() / window.TotalSeconds}";
        var prevKey = $"rate:{clientId}:{(now.ToUnixTimeSeconds() / window.TotalSeconds) - 1}";

        var currentCount = await _cache.GetAsync<int>(currentKey);
        var prevCount = await _cache.GetAsync<int>(prevKey);

        // Calculate weighted count
        var windowProgress = (now.ToUnixTimeSeconds() % window.TotalSeconds) / window.TotalSeconds;
        var weightedCount = prevCount * (1 - windowProgress) + currentCount;

        if (weightedCount >= maxRequests)
            return false;

        await _cache.IncrementAsync(currentKey);
        return true;
    }
}

Key Points to Mention

  • Use Redis INCR for atomic operations
  • Return appropriate HTTP headers (X-RateLimit-Remaining, Retry-After)
  • Consider graceful degradation vs hard blocking
  • Implement circuit breaker for rate limiter itself

How do you balance technical debt with feature delivery?

Framework Answer

1. Quantify Technical Debt:

  • Track debt items in backlog with estimated remediation cost
  • Measure impact: build times, bug rates, developer velocity
  • Use code metrics: cyclomatic complexity, test coverage trends

2. Categorize and Prioritize:

  • Critical: Security vulnerabilities, data integrity risks - fix immediately
  • High: Blocks feature development - plan within current sprint
  • Medium: Slows development - allocate 20% of sprint capacity
  • Low: Cosmetic/minor - opportunistic cleanup

3. Communication Strategy:

  • Present debt in business terms: "This will slow feature X by 3 weeks"
  • Create technical debt dashboard visible to stakeholders
  • Propose debt budget: "20% of each sprint for debt reduction"

4. Prevention:

  • Definition of Done includes debt consideration
  • Architecture reviews for significant changes
  • Automated quality gates in CI/CD
How would you approach modernizing a legacy monolithic application?

Phased Modernization Approach

Phase 0: Assessment (2-4 weeks)

  • Document current architecture and dependencies
  • Identify bounded contexts using Domain-Driven Design
  • Assess code quality, test coverage, deployment complexity
  • Interview team about pain points

Phase 1: Foundation (4-8 weeks)

  • Implement API facade in front of monolith
  • Set up CI/CD pipeline
  • Add monitoring and observability
  • Containerize monolith (lift and shift)

Phase 2: Incremental Extraction

  • Start with least coupled, highest-value domain
  • Extract to microservice with own database
  • Use anti-corruption layer for data sync
  • Route traffic gradually (feature flags)

Phase 3: Data Migration

  • Implement Change Data Capture (CDC)
  • Run dual-write during transition
  • Validate data consistency
  • Cut over when confident
Key Point: Never recommend "big bang" rewrites. Emphasize incremental, risk-managed migration with continuous delivery throughout.

Technical Deep-Dive Questions

Explain the differences between eventual consistency and strong consistency. When would you use each?

Strong Consistency

All nodes see the same data at the same time. After a write completes, all subsequent reads return that value.

Use When:

  • Financial transactions (account balances must be accurate)
  • Inventory management (can't oversell)
  • User authentication (security tokens)

Trade-offs: Higher latency, reduced availability during partitions

Eventual Consistency

Given enough time without new updates, all replicas will converge to the same value.

Use When:

  • Social media feeds (delayed posts acceptable)
  • Product catalog (brief inconsistency tolerable)
  • Analytics and reporting
  • Shopping cart (merge on checkout)

Trade-offs: Lower latency, higher availability, but complexity in conflict resolution

How would you implement authentication and authorization in a microservices architecture?

Recommended Approach: OAuth 2.0 + JWT

Authentication Flow:

  1. Client authenticates with Identity Provider (Azure AD, Auth0)
  2. IdP issues JWT access token and refresh token
  3. Client includes JWT in Authorization header for all requests
  4. API Gateway validates token signature and expiry
  5. Services extract claims from JWT for authorization

        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,

Before Architecture Reviews / Technical Discussions

  • Review context: stakeholders, constraints, existing systems
  • Prepare 3-5 concrete examples from your experience
  • Practice explaining complex designs simply (45 minutes)
  • Prepare clarifying questions and alternatives
  • Test video/audio if remote discussion
  • Have drawing tool ready (Excalidraw, paper)

During System Design

  • Clarify requirements before designing (5 min)
  • State assumptions explicitly
  • Do back-of-envelope calculations
  • Start with high-level, then deep dive
  • Discuss trade-offs for every decision
  • Address scalability and failure modes

Key Phrases to Use

  • "Let me clarify the requirements first..."
  • "The trade-off here is between X and Y..."
  • "Given our scale requirements, I would choose..."
  • "One alternative approach would be..."
  • "To handle failure scenarios, we could..."
