Introduction and Overview

What is PEGA BPM?

PEGA BPM (Business Process Management) is a low-code application development platform developed by Pegasystems. It enables organizations to build enterprise applications that automate business processes, manage cases, and deliver personalized customer experiences.

Interview Preparation

Beginner Level Questions

Q1: What is PEGA and what are its main components?

PEGA is a low-code Business Process Management (BPM) and Customer Relationship Management (CRM) platform.

Main Components:

  • Process Automation: Workflow and case management
  • Decision Management: Business rules engine
  • Customer Engagement: Omni-channel interactions
  • AI/ML: Predictive analytics and decisioning
  • Robotic Automation: RPA capabilities
Q2: Explain the difference between a Flow and a Case in PEGA.
Case Flow
Represents a unit of work Defines sequence of steps
Has lifecycle with stages Part of case implementation
Contains data, documents, history Contains shapes (assignments, decisions)
Can have child cases Can be reused across cases
Q3: What are the different types of properties in PEGA?

By Structure:

  • Single Value: One value (e.g., CustomerName)
  • Value List: Ordered collection (e.g., PhoneNumbers[])
  • Value Group: Named properties (e.g., Address.Street)
  • Page: Complex object (e.g., Customer)
  • Page List: Collection of objects (e.g., OrderItems[])
  • Page Group: Named collection of pages

By Data Type: Text, Integer, Decimal, Boolean, Date, DateTime, Time, Identifier, Password

Q4: What is a Data Page and why is it important?

A Data Page is PEGA's mechanism for loading, caching, and managing data from various sources.

Importance:

  • Performance: Caches data to reduce database calls
  • Reusability: Define once, use everywhere
  • Abstraction: Separates data access from UI/process
  • Flexibility: Multiple source options (DB, API, activity)

Key Properties: Scope (Thread/Requestor/Node), Structure (Page/List), Source, Refresh Strategy

Intermediate Level Questions

Q5: How does PEGA handle parallel processing in workflows?

PEGA supports parallel processing through:

  • Split-Join (Flow): Split shape creates parallel branches, Join waits for completion
  • Parallel Stages (Case): Multiple stages execute simultaneously
  • Child Cases: Create multiple child cases with propagation options
  • Spin-off Processing: Queue-based parallel execution
Q6: How do you implement error handling in PEGA integrations?

Integration Error Handling Approaches:

  • Connector-Level: Configure timeout, retry with exponential backoff, circuit breakers
  • Data Page Level: Handle "Object not found", configure fallback sources
  • Activity-Level: Try-Catch blocks, step-level error handling
  • Flow-Level: Ticket for exceptions, SLA-based escalation

Error Response Mapping:

  • HTTP 4xx â†’ Client error â†’ User notification
  • HTTP 5xx â†’ Server error â†’ Retry then escalate
  • Timeout â†’ Retry with exponential backoff

Q7: Explain PEGA's rule resolution algorithm.

Rule Resolution Priority Order:

  1. Circumstance/Specialization: Most specific circumstances win
  2. Class Hierarchy: Most specific class wins (child overrides parent)
  3. Ruleset List Order: Application â†’ Framework â†’ Pega
  4. Ruleset Version: Higher version number wins
  5. Circumstance Date/Time: Currently valid rules only
  6. Availability: Available rules only (Blocked/Withdrawn excluded)
